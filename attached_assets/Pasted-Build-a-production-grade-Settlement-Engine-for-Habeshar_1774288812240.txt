Build a production-grade Settlement Engine for Habeshare, including backend services, Firestore schema, admin APIs, and admin UI integration.

Important context:
Habeshare is a fintech / remittance platform with:
- Node.js + TypeScript backend
- Express APIs
- Firebase Admin SDK
- Firestore
- Existing modules already built:
  - walletService
  - treasuryService
  - paymentRouter
  - payout connectors
  - fraud monitoring
  - reconciliation service
  - admin console
  - FX marketplace
  - liquidity reservations
  - payout_transactions
  - ledger_entries
  - audit logs

Goal:
Implement the final missing core component: Settlement Engine.

This engine must track what Habeshare owes partners, what partners owe Habeshare, and the lifecycle of settlement and rebalancing after payouts are executed.

Do NOT break existing payout logic.

Responsibilities:

   - create settlement obligations after completed payouts
   - update obligation status
   - fetch settlement summaries

2. settlementLedgerService.ts
   - immutable settlement movement records
   - record opening, closing, rebalancing, adjustments

3. settlementBatchService.ts
   - group open obligations into daily or provider-based settlement batches
   - mark batches as processing, settled, failed

4. settlementReconciliationService.ts
   - compare partner-reported settlement vs internal obligations
   - flag mismatches

5. settlementAlertsService.ts
   - generate alerts for overdue settlement, mismatch, negative partner exposure

3) SETTLEMENT LOGIC
==================================================

Implement this lifecycle:

A. After a payout transaction becomes COMPLETED:
- create a settlement obligation
- direction depends on business model:
  usually OWED_TO_PARTNER for payout execution
- store provider, currency, amount, txId reference
- write settlement movement: SETTLEMENT_OPENED
- write audit log event: settlement_opened

B. Batch open obligations:
- batch by provider + currency + day
- create settlement_batch
- move obligations from OPEN -> BATCHED
- assign batchId
- write movement: SETTLEMENT_BATCHED

C. Mark batch as processing:
- status PROCESSING when settlement initiated

D. Mark batch as settled:
- mark batch SETTLED
- mark obligations SETTLED
- set settledAt
- write movement: SETTLEMENT_SETTLED
- write audit log event: settlement_completed

E. If settlement fails:
- mark batch FAILED
- mark obligations FAILED or return to OPEN based on retry policy
- generate settlement alert
- write movement: SETTLEMENT_FAILED

Requirements:

1. When payout succeeds:
   - confirm reservation if not already confirmed
   - create settlement obligation

2. When payout fails:
   - do NOT create settlement obligation
   - release treasury reservation as normal

3. Settlement engine must not duplicate obligations for the same txId

4. Add idempotency protection so repeated callbacks or retries do not create duplicate obligations
